## The "Final Boss" of Data Extraction
You’ve mastered `Nokogiri`. You can navigate a nested `<div>` structure with your eyes closed. But then, your client or your boss sends you a link to a folder containing 500 PDF invoices, reports, or government filings.

**PDFs are not data formats. They are visual formats.**

A PDF doesn't know what a "table" is. It only knows that the character "8" should be placed at coordinates `(x: 150, y: 400)`. When you try to scrape a PDF, you aren't parsing a tree; you're reconstructing a crime scene from scattered clues.

Here is how to use Ruby to pull structured data out of the chaos.

---

## 1. The "Digital Native" Approach: `pdf-reader`
If the PDF was generated by a computer (not scanned from a printer), the text is usually selectable. For these, the `pdf-reader` gem is your first line of defense.

```ruby
gem install pdf-reader
```

```ruby
require 'pdf-reader'

reader = PDF::Reader.new("invoice.pdf")

reader.pages.each do |page|
  # This returns a massive, often messy string of all text on the page
  text = page.text
  puts text
end
```

### The Problem: The "Jumble"
When you run `page.text`, the layout often breaks. Columned data might get merged, or headers might appear in the middle of sentences. To fix this, we need **Pattern Matching**.

---

## 2. Cleaning the Mess with Regex
Since we can't use CSS selectors, we have to use Ruby's powerful String and Regex methods to find our "anchors."

Imagine an invoice that looks like this: `Total Amount: $1,250.00`.

```ruby
text = page.text

# Use a named capture group to find the price after the anchor text
if (match = text.match(/Total Amount:\s+\$(?<price>[\d,.]+)/))
  puts "Found Price: #{match[:price]}"
end
```

**Pro Tip:** Use the `.scan` method if you need to extract multiple items (like rows in a table) that follow a consistent pattern.

---

## 3. Handling Tables: `pdf-reader` vs. Layout
If the PDF has a complex table, `page.text` often fails because it reads line-by-line across the whole page. 

For these, you need to use **coordinate-based extraction**. The `pdf-reader` gem allows you to inspect the raw "walk" of the PDF to see where characters are placed. However, a more "Ruby-way" to handle this is to use a wrapper around the **Poppler** library (via the `poppler` gem) which is much better at preserving layouts.

---

## 4. The "Nuclear Option": OCR with Tesseract
What if the PDF is just a series of images (a scan)? `pdf-reader` will return an empty string. You need **Optical Character Recognition (OCR)**.

First, install Tesseract on your system (`brew install tesseract` or `pacman -S tesseract`). Then, use the `rtesseract` gem.

```ruby
require 'rtesseract'

# Convert the PDF page to an image first (using MiniMagick)
# Then process the image
image = RTesseract.new("scanned_page.png")
puts image.to_s # The extracted text
```

---

## 5. The Workflow: A Professional Pipeline
When I have to scrape "messy" PDFs at scale, I build a pipeline:

1.  **Categorize:** Use `pdf-reader` to check if there is any embedded text.
2.  **Extract:** If digital, use `page.text`. If scanned, send to an OCR worker.
3.  **Sanitize:** Use `text.gsub!(/\s+/, ' ')` to normalize white space.
4.  **Parse:** Use a series of Regex "rules" to populate a Ruby Hash.
5.  **Validate:** Check if the extracted "Total" matches the sum of the "Line Items." If not, flag the PDF for manual review.

---

## 6. Summary: Choose Your Tool

| Scenario | Tool |
| :--- | :--- |
| **Clean, digital PDF** | `pdf-reader` + Regex |
| **Complex Tables** | `tabula-java` (via Ruby wrapper) |
| **Scanned/Images** | `rtesseract` (OCR) |
| **Batch Processing** | `combine_pdf` (to split/merge before parsing) |

## Final Thought
Scraping PDFs is a game of "good enough." You will rarely get 100% accuracy on the first try. Focus on finding the **unique anchors** (like Account Numbers or Tax IDs) and use Ruby’s string manipulation to do the heavy lifting.
